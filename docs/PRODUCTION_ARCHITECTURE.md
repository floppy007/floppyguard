# FloppyGuard Production Architecture

Stand: 2026-05-03

## Overview

FloppyGuard runs host-based on a single server. There is no Docker container for the application itself.

```
Internet → nginx (80/443) → proxy host configs in /data/nginx/
                          → port 81 (admin UI, SPA + API proxy)

systemd → floppyguard-backend (Node.js, port 3300)
MySQL   → npm database @ localhost:3306
```

## Systemd Service

**Unit file:** `/etc/systemd/system/floppyguard-backend.service`

Key settings:
- `WorkingDirectory`: `/var/www/nginx-proxy-manager-fork-base/backend`
- `ExecStart`: `/bin/node index.js`
- `PORT=3300`, `NODE_ENV=production`
- `DB_MYSQL_HOST=localhost`, `DB_MYSQL_USER=npm`, `DB_MYSQL_PASSWORD=npm`, `DB_MYSQL_NAME=npm`
- `NPM_LETSENCRYPT_ROOT=/opt/npm/letsencrypt`
- `NPM_CUSTOM_SSL_ROOT=/opt/npm/data/custom_ssl`
- `Restart=on-failure`, `RestartSec=5`

Commands:
```bash
systemctl start|stop|restart|status floppyguard-backend
journalctl -u floppyguard-backend -f
```

## Nginx Configuration

**Config file:** `/etc/nginx/conf.d/floppyguard.conf`

What it does:
- Port 81: serves `frontend/dist` as SPA (try_files → index.html), proxies `/api/` to `127.0.0.1:3300`
- Port 80 default: catch-all fallback with ACME challenge support
- Port 443 default: rejects unknown SNI
- Includes `/data/nginx/**/*.conf` — proxy host configs generated by the backend

**Include files:** `/etc/nginx/conf.d/include/`
- Copied from legacy NPM Docker image on cutover
- Contains: block-exploits, ssl-ciphers, ssl-cache, proxy, assets, force-ssl, letsencrypt-acme-challenge, resolvers, ip_ranges, log-proxy

After any config change:
```bash
nginx -t && nginx -s reload
```

## Data Directories

| Path | Contents |
|------|----------|
| `/data/` | FloppyGuard runtime data root |
| `/data/nginx/` | Generated nginx configs (proxy_host, redirection_host, dead_host, stream, temp) |
| `/data/logs/` | nginx access/error logs per host |
| `/data/letsencrypt-acme-challenge/` | ACME http-01 challenge files |
| `/data/keys.json` | JWT signing keys |
| `/data/access/` | Access list htpasswd files |
| `/data/custom_ssl/` | Custom SSL certificate files |

## SSL Certificates

Certificates are stored at `/opt/npm/letsencrypt/` (legacy NPM volume path).

Symlinks into system certbot paths:
```
/etc/letsencrypt/live    → /opt/npm/letsencrypt/live
/etc/letsencrypt/archive → /opt/npm/letsencrypt/archive
```

Certbot plugins (Cloudflare, dns-multi) are installed in a Python virtualenv:
```
/opt/certbot/bin/activate
/opt/certbot/bin/pip
/opt/certbot/bin/certbot
```

The backend calls `. /opt/certbot/bin/activate && pip install ...` on startup to ensure plugins are current.

## Port Map

| Port | Process | Purpose |
|------|---------|---------|
| 80 | nginx | HTTP proxy hosts + ACME challenges |
| 81 | nginx | FloppyGuard admin UI |
| 443 | nginx | HTTPS proxy hosts |
| 3300 | floppyguard-backend | API (localhost only) |
| 3306 | mysqld | Database |
| 8080 | nginx | wg-gui (legacy, pending removal) |

## Startup Order

1. MySQL must be running (backend waits for DB connection)
2. `systemctl start floppyguard-backend` — runs migrations, certbot plugin setup, starts Express
3. `nginx -s reload` — picks up any config changes written by the backend

Systemd `After=mysql.service mariadb.service` handles ordering on boot.

## nginx Config Lifecycle

When a proxy host is created/modified in the UI:
1. Backend writes `/data/nginx/proxy_host/<id>.conf`
2. Backend calls `nginx -t` (config test)
3. If OK: `nginx -s reload`
4. If not OK: config is renamed to `.err`, host marked offline in DB

## Legacy NPM Data

The legacy NPM Docker container (`/opt/npm/`) was stopped as part of cutover. Its data remains:
- `/opt/npm/data/` — original NPM data (not used by FloppyGuard)
- `/opt/npm/letsencrypt/` — certificates (still in use via symlinks)
- `/opt/npm/mysql/` — MariaDB data (legacy, the live DB is the host MySQL)

The MariaDB Docker container (`npm-db-1`) can be stopped if the host MySQL holds the primary data.
